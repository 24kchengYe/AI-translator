使用MCP进行代码执行：构建更高效的代理

发布2025年11月04日

直接调用工具会为每个定义和结果消耗上下文。代理通过编写代码来调用工具，从而更好地扩展。以下是它与MCP的工作原理。

模型上下文协议（MCP）是一个连接AI代理与外部系统的开放标准。将代理连接到工具和数据通常需要为每个配对进行自定义集成，这导致碎片化和重复工作，使得真正连接的系统难以扩展。MCP提供了一个通用协议——开发者在其代理中实现MCP一次，就可以解锁整个集成生态系统。

自2024年11月推出MCP以来，采用速度非常快：社区已经构建了数千个MCP服务器，所有主要编程语言都有SDK可用，行业已将MCP作为连接代理与工具和数据的事实标准。今天，开发者通常构建可以访问数百或数千个工具的代理，跨越数十个MCP服务器。然而，随着连接工具数量的增加，预加载所有工具定义并通过上下文窗口传递中间结果会减慢代理的速度并增加成本。

在这篇博客中，我们将探讨代码执行如何使代理更高效地与MCP服务器交互，同时使用更少的令牌处理更多工具。

工具的过度令牌消耗使代理效率降低

随着MCP使用的扩展，有两种常见模式可能会增加代理的成本和延迟：

工具定义超载上下文窗口；

  1. 中间工具结果消耗额外的令牌。
  2. 1. 工具定义超载上下文窗口

大多数MCP客户端将所有工具定义直接加载到上下文中，使用直接的工具调用语法将它们暴露给模型。这些工具定义可能看起来像：

gdrive.getDocument 描述：从Google Drive检索文档 参数： documentId（必需，字符串）：要检索的文档ID fields（可选，字符串）：要返回的特定字段 返回：包含标题、正文内容、元数据、权限等的文档对象
salesforce.updateRecord 描述：更新Salesforce中的记录 参数： objectType（必需，字符串）：Salesforce对象的类型（潜在客户、联系人、账户等） recordId（必需，字符串）：要更新的记录ID data（必需，对象）：要更新的字段及其新值 返回：带有确认的更新记录对象

工具描述占用更多的上下文窗口空间，增加响应时间和成本。在代理连接到数千个工具的情况下，他们需要处理数十万个令牌才能读取请求。

2. 中间工具结果消耗额外的令牌

大多数MCP客户端允许模型直接调用MCP工具。例如，您可能会问您的代理：“从Google Drive下载我的会议记录并将其附加到Salesforce潜在客户。”

模型将进行如下调用：

工具调用：gdrive.getDocument(documentId: "abc123") → 返回“讨论了Q4目标...\n[完整的记录文本]”（加载到模型上下文中） 工具调用：salesforce.updateRecord( objectType: "SalesMeeting", recordId: "00Q5f000001abcXYZ", data: { "Notes": "讨论了Q4目标...\n[完整的记录文本]" } ) （模型需要再次将整个记录写入上下文）

每个中间结果必须经过模型。在这个例子中，完整的调用记录流经两次。对于一个2小时的销售会议，这可能意味着处理额外的50,000个令牌。即使是更大的文档也可能超过上下文窗口限制，打断工作流程。

对于大型文档或复杂数据结构，模型在工具调用之间复制数据时可能更容易出错。

使用MCP的代码执行提高了上下文效率

随着代码执行环境在代理中变得越来越普遍，一个解决方案是将MCP服务器呈现为代码API，而不是直接的工具调用。代理可以编写代码与MCP服务器交互。这种方法解决了两个挑战：代理可以仅加载所需的工具，并在执行环境中处理数据，然后将结果传回模型。

Cloudflare发布了类似的发现，将与MCP的代码执行称为“代码模式”。核心见解是相同的：LLMs擅长编写代码，开发人员应该利用这一优势来构建与MCP服务器更高效交互的代理。

与MCP的代码执行的好处

与MCP的代码执行使代理能够通过按需加载工具、更有效地过滤数据在到达模型之前，以及在单一步骤中执行复杂逻辑来更有效地使用上下文。使用这种方法还有安全性和状态管理的好处。

渐进式披露

模型在导航文件系统方面表现出色。将工具作为文件系统中的代码呈现，使模型能够按需读取工具定义，而不是一次性读取所有定义。

或者，可以搜索工具工具可以添加到服务器以查找相关定义。例如，当使用上述假设的Salesforce服务器时，代理搜索“salesforce”并仅加载当前任务所需的工具。在搜索工具工具中包含一个详细级别参数，允许代理选择所需的详细程度（例如仅名称、名称和描述，或带有模式的完整定义），也有助于代理节省上下文并有效找到工具。

上下文高效的工具结果

在处理大型数据集时，代理可以在返回结果之前在代码中过滤和转换结果。考虑获取一个10,000行的电子表格：

// 无代码执行 - 所有行都通过上下文流动

代理看到五行而不是10,000行。类似的模式适用于聚合、跨多个数据源的连接或提取特定字段——所有这些都不会膨胀上下文窗口。

更强大且上下文高效的控制流

循环、条件语句和错误处理可以使用熟悉的代码模式完成，而不是链接单个工具调用。例如，如果您需要在Slack中发送部署通知，代理可以编写：

let found = false;

这种方法比在代理循环中交替进行MCP工具调用和睡眠命令更有效。

此外，能够写出一个被执行的条件树也可以节省“首次令牌”的延迟：代理可以让代码执行环境来处理，而不必等待模型评估一个if语句。

保护隐私的操作

当代理使用MCP进行代码执行时，中间结果默认保留在执行环境中。这样，代理只看到你明确记录或返回的内容，这意味着你不希望与模型共享的数据可以在工作流中流动，而不会进入模型的上下文。

对于更敏感的工作负载，代理工具可以自动对敏感数据进行标记。例如，假设你需要从电子表格中导入客户联系信息到Salesforce。代理写道：

const sheet = await gdrive.getSheet({ sheetId: 'abc123' }); for (const row of sheet.rows) { await salesforce.updateRecord({ objectType: 'Lead', recordId: row.salesforceId, data: { Email: row.email, Phone: row.phone, Name: row.name } }); } console.log(`更新了 ${sheet.rows.length} 个潜在客户`);

MCP客户端在数据到达模型之前拦截数据并对PII进行标记：

// 如果代理记录了sheet.rows，它会看到： [ { salesforceId: '00Q...', email: '[EMAIL_1]', phone: '[PHONE_1]', name: '[NAME_1]' }, { salesforceId: '00Q...', email: '[EMAIL_2]', phone: '[PHONE_2]', name: '[NAME_2]' }, ... ]

然后，当数据在另一个MCP工具调用中共享时，它通过MCP客户端的查找进行解标记。真实的电子邮件地址、电话号码和姓名从Google Sheets流向Salesforce，但从未经过模型。这防止了代理意外记录或处理敏感数据。你还可以用它来定义确定性的安全规则，选择数据可以流向何处和从何处流出。

状态持久性和技能

具有文件系统访问权限的代码执行允许代理在操作之间保持状态。代理可以将中间结果写入文件，使它们能够恢复工作并跟踪进度：

const leads = await salesforce.query({ query: 'SELECT Id, Email FROM Lead LIMIT 1000' }); const csvData = leads.map(l => `${l.Id},${l.Email}`).join('\n'); await fs.writeFile('./workspace/leads.csv', csvData); // 后续执行从中断处继续

代理还可以将自己的代码持久化为可重用的函数。一旦代理为某个任务开发出可工作的代码，它可以保存该实现以供将来使用：

// 在 ./skills/save-sheet-as-csv.ts 中导入所有 gdrive 从 './servers/google-drive'; 导出异步函数 saveSheetAsCsv(sheetId: string) { const data = await gdrive.getSheet({ sheetId }); const csv = data.map(row => row.join(',')).join('\n'); await fs.writeFile(`./workspace/sheet-${sheetId}.csv`, csv); return `./workspace/sheet-${sheetId}.csv`; } // 后面，在任何代理执行中：导入 { saveSheetAsCsv } 从 './skills/save-sheet-as-csv'; const csvPath = await saveSheetAsCsv('abc123');

这与概念紧密相关技能，可重用指令、脚本和资源的文件夹，以提高模型在专业任务上的性能。向这些保存的函数添加 SKILL.md 文件创建了一个结构化的技能，模型可以引用和使用。随着时间的推移，这允许你的代理构建一个更高层次能力的工具箱，发展它最有效工作的支架。

请注意，代码执行引入了自身的复杂性。运行代理生成的代码需要一个安全的执行环境，具备适当的沙箱，资源限制和监控。这些基础设施要求增加了操作开销和安全考虑，而直接工具调用则避免了这些。代码执行的好处——降低的令牌成本、更低的延迟和改善的工具组合——应与这些实施成本进行权衡。

摘要

MCP 为代理连接到许多工具和系统提供了基础协议。然而，一旦连接的服务器过多，工具定义和结果可能消耗过多的令牌，从而降低代理效率。

尽管这里的许多问题感觉新颖——上下文管理、工具组合、状态持久性——但它们在软件工程中有已知的解决方案。代码执行将这些既定模式应用于代理，让它们使用熟悉的编程结构更有效地与 MCP 服务器交互。如果你实施这种方法，我们鼓励你与MCP 社区分享你的发现。
